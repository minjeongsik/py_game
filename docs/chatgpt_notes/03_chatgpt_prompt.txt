아래는 내가 만들고 있는 C# / MonoGame 프로젝트에 대한 정리입니다.
이 내용을 바탕으로 현재 구조 진단, 리팩터링 우선순위, 다음 개발 방향을 구체적으로 조언해줘.

[프로젝트 개요]
- 프로젝트명: Aether Trail
- 기술 스택: C# / .NET 8 / MonoGame
- 장르: 2D 크리처 수집형 RPG
- 현재 목표: 완성형 게임이 아니라 vertical slice를 만드는 중

[현재 플레이 루프]
- Title 메뉴 진입
- New Game 또는 Continue
- 월드 맵 탐험
- 특정 지역에서 랜덤 인카운터 발생
- 턴 기반 전투 진행
- Attack / Item / Capture / Run 처리
- 포획 성공 시 파티 또는 스토리지 반영
- 저장/불러오기 가능

[현재 폴더 구조]
- Program.cs: 게임 실행 진입점
- Game1.cs: 메인 게임 루프, 상태 전환, 월드/전투/UI 흐름 대부분 담당
- Core: 입력, 상태 관리, 카메라, 세션 등 공통 시스템
- World: 맵 로딩, 렌더링, 이동, 포탈, 랜덤 인카운터
- Battle: 전투 컨트롤러
- Creatures: 크리처 종 정의 및 개체 데이터
- Data: JSON 로딩, 저장/불러오기, 데이터 정의
- UI: 픽셀 폰트 렌더링과 UI 관련 코드
- Content/Data: creatures, moves, items, zones, maps 등의 JSON 데이터

[현재 구현된 기능]
- 타이틀 메뉴
- 월드 이동과 충돌 처리
- 포탈 기반 지역 이동
- 랜덤 인카운터
- 턴 기반 1대1 전투
- 회복 아이템 / 포획 아이템 사용
- 파티 / 스토리지 관리
- 저장 / 불러오기
- 디버그 오버레이

[현재 구조에서 느끼는 문제]
- Game1.cs에 로직이 많이 몰려 있음
- 전투, 월드, UI, 상태 전환이 한 곳에 강하게 결합되어 보임
- 데이터는 JSON으로 분리되어 있는데 코드 구조는 아직 프로토타입 느낌이 강함
- 앞으로 기능이 늘어나면 유지보수가 어려워질 것 같음

[원하는 답변]
1. 현재 구조의 장점과 단점을 분석해줘.
2. 지금 단계에서 가장 작은 안전한 리팩터링 3가지를 우선순위로 제안해줘.
3. Game1.cs를 어떤 단위로 분리하면 좋은지 예시 구조를 제안해줘.
4. vertical slice 다음 단계에서 무엇을 먼저 확장해야 하는지 추천해줘.
5. 전투 / 월드 / 데이터 / UI 중 어디를 먼저 손대는 게 가장 효율적인지 이유와 함께 말해줘.
6. 가능하면 폴더 구조 예시와 클래스 책임 분리 예시도 보여줘.
