# Future Work Guide

이 저장소에서 다시 작업을 이어갈 때 참고할 수 있도록, 현재까지의 작업 방식과 기대되는 흐름, 주요 체크리스트를 정리했습니다. 새로운 의뢰를 수행할 때 다음 순서를 따라주세요.

## 1. 검사 / 환경 점검
- `AGENTS.md`을 항상 먼저 읽고 빌드/검증/보고 규칙을 숙지하세요.
- `DETAIL_*` 체크리스트(특히 `DETAIL_UX_POLISH_CHECKLIST.md`)와 `MASTER_RUNTIME_IMPLEMENTATION_CHECKLIST.md`/`DETAIL_GAMEPLAY_CHECKLIST.md` 등의 단계별 체크리스트를 항상 최신 상태로 확인하고, 작업이 어느 단계에 해당하는지 파악하세요.
- `git status`로 작업 디렉터리를 깨끗하게 유지하고, `dotnet build PyGame.sln`을 프로젝트 변경 또는 빌드 영향을 주는 콘텐츠 변경 직후 실행하세요.

## 2. 작업 흐름
1. **문제/요구 분석**: 체크리스트에서 요구하는 항목을 정리하고 필요한 화면(UX), 시스템(맵/전투 등)을 파악합니다.
2. **코드 변경**: 변경은 `Core`, `GameFlow`, `UI` 등 구성 요소를 최소한으로 건드리며, `UiSkinRenderer`, `PixelTextRenderer` 등을 재사용하세요. 변경 전에 어떤 파일을 건드릴지 명확히 설명하는 커멘트를 남기면 좋습니다.
3. **자산/그림**: 새 UI/젠더/타일 리소스를 추가할 때는 `Content/` 하위에 넣고 `GameDefinition` 등에서 참조하도록 정리합니다.
4. **QA 및 빌드**: `dotnet build PyGame.sln`을 수행하고 출력 결과를 기록합니다. 실패하면 원인과 해결 방안을 포함해 보고하세요.
5. **워크 리포트 작성**: `docs/work_reports/`에 보고서를 남기며, 다음 항목을 포함하세요.
   - Summary
   - Files changed
   - Checklist 항목 반영 사항
   - Build command/result
   - Remaining issues

## 3. UX/GUI 체크리스트 대응
- 개선 항목은 `DETAIL_UX_POLISH_CHECKLIST.md`에 정의된 순서(대화/버튼/문구 정렬 → 입력 안내 → 글자/폰트 → UI 패널 → 안내/목록 정렬)를 따라 작업하세요.
- 각 항목을 반영할 때 말머리로 “UX 체크리스트”를 포함하고, 리포트에 체크 상태와 변경 내용을 명시합니다.
- 설명 박스나 힌트 패널은 맵 등과 겹치지 않게 배치해 지도 시야를 확보하고, 필요한 경우 위치/크기/투명도 등을 사례별로 조정하며 문서화하세요.

## 4. 맵/카메라/타일 체크리스트 준비
- 맵 수정 단계에서는 타일셋, 카메라 트랜스폼, 해상도, UI 배치 등을 함께 점검하세요.
- 좌측 상단 힌트 박스처럼 맵을 덮는 요소는 우측으로 이동시키거나 맵 외부 여백에 붙여 맵 콘텐츠를 가리지 않도록 합니다.
- 타일/맵 데이터(`Content/Definitions/maps/` 내 JSON 등)를 건드릴 때는 `GameDefinitionValidator`를 통과시키고, 필요한 경우 `Runtime` 설정도 갱신하세요.

## 5. 커뮤니케이션/추적
- 매 작업마다 `docs/work_reports/` 아래에 새로운 리포트를 추가하고, 작업 요약과 QA/빌드 결과를 반드시 포함합니다.
- 체크리스트 항목을 실제로 끝낸 뒤에는 문서 안에 `[x] (완료일: YYYY-MM-DD)` 형태로 체크 표시를 추가하며, 완료일을 채웁니다.
- 질문이나 결정이 필요한 부분은 `communication/` 디렉터리와 이 가이드 파일에 이어서 브레인스토밍 노트를 남겨두면 흐름을 이어받기 쉽습니다.

## 6. 기타
- 자주 쓰는 단축키/입력 안내(`Enter`, `Esc`, `I`, 방향키 등)를 공통 `InputHintRenderer` 등으로 관리해 재사용하세요.
- 작업 중 `world`, `battle`, `business` 등과 관련한 그래픽/패널이 서로 어울리도록 일관된 UI 스킨과 텍스트 스타일을 유지합니다.
- 작업이 끝나면 사용자에게 간단한 사용자 업데이트(무슨 작업을 했고 다음 단계가 무엇인지)와 함께 `docs/work_reports/` 결과를 공유하세요.

## 7. 체크리스트/QA 상태 표
- `DETAIL_*` 체크리스트와 `MASTER_RUNTIME_IMPLEMENTATION_CHECKLIST.md`에서 현재 작업 중인 섹션을 하나의 표로 기준 날짜 및 상태(예: 진행 중, 대기, 완료일)를 정리해 두면 빠르게 우선순위를 잡을 수 있습니다. 예: `| 항목 | 상태 | 완료일 | 비고 |`.
- QA/빌드 반복시에는 항상 `dotnet build PyGame.sln` 출력과 함께 `tmp_pause_view.txt`처럼 시점별 화면 캡처 결과를 기록하세요. 캡처파일은 같은 `tmp_*.txt` 또는 `tmp_*.png` 형식을 유지해 후속 작업자가 찾기 쉽도록 합니다.

## 8. 명명/텍스트/톤 표준
- 한국어 텍스트는 자연스럽고 짧게 유지합니다. UI 타이틀은 3~6자 내외, 설명/상태 메시지는 12~28자 안에서 한 줄 혹은 두 줄로 요약합니다.
- 공통 입력 안내는 `Enter`, `Esc`, `방향키`, `WASD`, `I` 등으로 고정 표현하며 괄호/비속어 없이 둡니다. 버튼/창 제목은 명사형, 설명은 명령형(“고르세요”, “눌러서”) 어조를 유지하세요.
- 필요한 경우 `docs/work_reports/` 리포트 맨 아래에 해당 변경에 쓰인 단어/문구를 짧게 나열해 일관성을 참고할 수 있게 합니다.

## 9. FAQ
- **Q:** “디테일 체크리스트 순서는 꼭 지켜야 하나요?”  
  **A:** 현재 진행 흐름(UX → 게임플레이 → 맵/카메라 등)은 우선순위와 빌드 위험을 줄이기 위한 것이므로 되도록 그대로 따르고, 필요하면 각 단계에서 소규모 조정을 메모에 남겨 주세요.
- **Q:** “화면 캡처는 누가 찍어요?”  
  **A:** 변경마다 `tmp_pause_view.txt` 등 간단한 캡처를 수동으로 만들고, 결과를 `docs/work_reports/` 리포트에 링크처럼 언급합니다. 애셋 교체만 있었다면 `Content/Sprites/` 등 경로를 함께 표기하세요.
- **Q:** “내가 중간에 작업을 멈추면 어떻게 이어서 하나요?”  
  **A:** 이 가이드와 `docs/work_reports/` 최근 파일을 먼저 읽고, 체크리스트에서 `[x]`를 수정하지 않은 항목을 중심으로 힌트를 찾은 다음 빌드/QA를 반복하세요.

이 지침을 바탕으로 다음 기회에 바로 이어질 수 있게 준비해두었으니, 추가로 필요한 내용이 있으면 알려주세요.
